# write-prd 技能

## 概述

本技能用于编写两类文档，模板位于 `./template/`：

| 文档类型 | 模板文件 | 用途 |
|---------|---------|------|
| **产品概要说明书** | `product-overview-template.md` | 整个产品/模块的顶层说明，含9章 |
| **菜单需求文档** | `requirement-document-template.md` | 单个菜单/页面的详细需求，含固定 ### 章节骨架 |

**核心纪律：** 所有 `###` 章节标题必须与模板完全一致，不得增删改序。无内容的节写「不适用」并说明原因，不得留空。

## 工作流程

### Step 1：判断文档类型

若不明确，先询问用户是写**产品概要说明书**还是**菜单需求文档**。

### Step 2：收集必要信息

**产品概要说明书需要：**
- 产品/功能模块名称 & 目标
- 业务背景、痛点、涉及角色
- 系统主流程（端到端）
- 外部依赖系统
- 非功能需求约束

**菜单需求文档需要：**
- 菜单名称、功能目标、所属产品模块
- 涉及角色及数据/功能权限
- 页面结构（有无左树、卡片视图、锚点）
- 查询字段、列表字段、表单字段
- 是否含导入/导出
- 是否有流程（状态流转）
- 需要校验的字段（正则或业务规则）
- 枚举值全集

### Step 3：读取模板并生成文档

1. 读取对应模板全部内容
2. 按模板章节顺序逐节填写
3. 输出给用户确认
4. 确认后写入：`docs/current/modules/{模块}/PRD_{功能}.md`

## 一、产品概要说明书编写指南

### 通用规则

- 从**业务与能力**角度描述，禁止描述前后端实现、接口与代码结构
- 架构图节点一律为**业务域、能力群或使用主体**名称
- 术语表帮助读者对齐理解，避免歧义
- 字段名称列仅写中文业务名，英文 key 写在说明列

### 各章编写要点

| 章号 | 标题 | 关键要求 |
|:---|:---|:---|
| **一** | **概述** | 含 1.1 背景与现状（业务场景 + 核心问题列表 + 方案目标）、1.2 产品目标（可量化）、1.3 术语表 |
| **二** | **角色与权限** | 角色定义表（角色/职责/业务入口）+ 权限矩阵（操作 × 角色），页面级细矩阵可在菜单需求文档展开 |
| **三** | **系统整体架构与主流程** | 见下方详细说明 |
| **四** | **菜单与需求文档索引** | 菜单名称 → 需求文档路径映射表 |
| **五** | **非功能需求** | 性能/P95响应时间/并发容量、可用性/降级方案、安全/审计/合规、兼容性/存量数据、可维护性/日志监控、可测试性 |
| **六** | **验收标准** | 功能验收(AC编号)、关联能力变更验收、非功能需求验收（含测试方法） |
| **七** | **外部依赖** | 依赖系统/服务的用途、对接形态、对接状态 |
| **八** | **版本与变更记录** | 本版本相对上一版本的功能变化表 |
| **九** | **附录** | 业务异常与提示口径、审计事件清单等 |

#### 第三章「系统整体架构与主流程」编写要求

**3.1 系统总体架构：**
- 必须输出 Mermaid 架构图，分层示意：使用与访问层 → 核心业务层 → 共享业务能力层 → 外部协作
- 分层数量与命名可按项目裁剪，但节点必须是业务域/能力群名称
- 架构说明表逐层填写：使用主体类型、业务域与菜单对应关系、跨域复用能力、外部协作方

**3.2 系统级主流程：**
- 跨角色、跨模块的**主成功路径**（从触发到归档/闭环）
- 分支与异常以业务语言简述
- 格式：`[阶段一]：[谁、在什么业务条件下、完成什么业务动作]` → `[阶段二]` → `[终态]`

**3.3 外部协作关系（业务级）：**
- 表格列出外部协作方、业务依赖、平台对外输出、责任边界

## 二、菜单需求文档编写指南

### 通用规则

- **可验收**：交互与数据规则可验收；不要求颜色字体间距
- **禁止实现细节**：禁止写组件库名、标签、属性（如 `el-*`、`v-model`）；用「数据表格」「主按钮」「抽屉」「分页器」等
- **通俗易懂**：短句、术语与枚举表一致；概述用「用户动作 + 系统反馈」；分支用「若…则…」
- **字段表**：必须含字段名、类型、必填、默认值、是否唯一值、数据来源、说明。字段名列仅写中文业务名，英文 key 放在说明列（如「对应数据键：reportNo」）
- **编写要求优先级**：以本技能及模板中 `> **编写要求**` 注释为准

### 章节骨架（### 标题固定，不得增删改序）

#### 〇、文档信息

| 项目 | 内容 |
|------|------|
| 文档名称 | 与封面标题一致 |
| 文档版本 | 如 v1.0，须与版本目录名一致 |
| 状态 | 未确认 / 已确认 |
| 确认日期 | 已定稿时填写 |
| 存放路径 | `docs/current/modules/{模块}/PRD_{功能}.md` |

#### 功能概述
一句话说清目标与用途。与兄弟页分文档时一句说明分工。避免纯口号。

#### 角色权限
**必有**。数据权限（单位/区域/行业/全局）+ 功能权限（增删改查）。用维度表 + 矩阵呈现。无差异写「不适用」及理由。

#### 页面结构
**必有框线图**（ASCII 或 Mermaid）。覆盖主区、查询/工具栏/主内容/分页、主要交互。弹层标注打开方式。含左树时须同时满足 `### 目录树` 节。有锚点导航时另增 `#### 锚点明细清单`。

#### 枚举
静态码表全集，格式：`#### 枚举：<含义>` + 表格。表头用中文，首列「存储值」对应持久化/接口取值，次列「展示名」。无则写「不适用」并说明。

#### 目录树（有树页 · 条件强制）
须在「页面结构」体现布局，并给出：
- **层级结构图**（缩进树或 Mermaid）+ **节点字段表**（标识、父标识、显示名、排序、类型/是否叶子）
- 最大深度、多根、虚拟根、同级排序、父子表达
- 选中展开联动主区、懒加载触发条件与来源、树上增删改拖拽操作
- 默认展开/选中态、空树占位

#### 查询功能
查询字段表（列同通用规则「字段表」）。写清查询/重置/回车行为。

#### 列表展示
**至少一张**列表字段表，或「不适用」+ 理由。禁止仅写「支持分页排序」。多专题分多张表。

#### 列表卡片（条件强制）
有卡片/列表切换时**必有**：框线图 + 结构 + 交互 + 与列表关系。

#### 工具栏按钮
名称、主次、显隐条件、打开方式、操作结果。用「用户动作 + 系统反馈」描述。

#### 表单设计
字段表 + 校验。抽屉/侧滑写入口、分区、主操作/取消、未保存关闭确认。联合唯一写在说明中。无不适用须声明。

#### 流程图
**须同时使用** Mermaid 流程图与**编号步骤列表**（两者语义一致，便于评审与测试对齐）。写清可执行规则与失败提示。无不适用须声明。

#### 导入导出（含导入/导出时 · 条件强制）
- **仅 Excel**（`.xlsx`/`.xls`）；先确认字段再落盘
- 导入说明（模板列、失败行/部分成功）+ **导入字段表**
- 导出说明 + **导出字段表**
- 字段表至少含：名称、key、类型、必填（导入）、默认值、是否唯一值、数据来源、校验/错误提示（导入）、脱敏（导出）
- 错误提示与 `### 数据验证规则` 同一列不得两套文案
- 无导入/导出时写「不适用」及理由

#### 数据验证规则
按固定结构编写（见下方第三章），不得仅对话输出。

#### 注意事项
仅业务/交互/验收边界；不含文档路径、README/skill 名、文档管理条文、UI 像素与组件库样式。

## 三、数据验证规则结构（菜单需求文档专用）

以下结构必须严格遵守，子结构须落在 `### 数据验证规则` 下，不新增同级 `###`：

```
### 数据验证规则

#### 校验范围与场景
[描述哪些操作场景触发校验：新增、编辑、Excel 导入等]

#### 正则形态校验（按字段）
[每个字段一个 ##### 小节]

##### <中文字段名>（`fieldKey`）
  1. 适用场景：新增、编辑、Excel 导入
  2. 正则表达式：
     ```regex
     ^正则表达式$
     ```
  3. 错误提示：[面向用户的提示文案]

[第二字段同结构...]

#### 其它验证规则（非正则）
[编号或分段说明，不用大表]

#### 跨字段与业务规则（可选）
[涉及多字段组合的业务验证规则]

#### 规则汇总（验收清单）
[逐条列出的可验收校验规则清单]
```

**注意：** 无正则字段时，在 `#### 正则形态校验（按字段）` 下写「本页无正则校验字段」及理由。
